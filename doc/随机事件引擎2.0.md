## 一、Game Over 条件（失败终止逻辑）

> **这是引擎的前提**。必须在写事件系统之前确定，否则开发 `simulator.ts` 时会踩坑。

### 1.2 各条件详细定义

#### 条件 A：现金破产（CASH_BANKRUPT）
```
真实逻辑：现金 <= 0 本身不等于死亡。
正常经营中，1 个月的现金流为负很常见（欠款在路上）。
真正的破产是："现金为负 + 未来 4 周内没有任何应收款到账"。

触发流程：
  Week 1（cash <= 0）：出现黄色预警，检查应收款
    → 若 4 周内有应收款 >= |cash|：标记为"危机但可救"，继续运行
    → 若无应收款兜底：开始计 cashCriticalStreak

  cashCriticalStreak 到达 4（即持续 1 个月）：
    → 最后警告，AI 给出"最后自救选项"（融资/裁员/变现资产）
    → 下一周仍未解决：触发 Game Over（CASH_BANKRUPT）
```

```typescript
// src/engine/game-over.ts

interface Receivable {
  amount: number       // 应收金额
  dueInWeeks: number   // 预计到账周数（相对当前）
}

function checkCashBankrupt(state: GameState): boolean {
  if (state.company.cash > 0) {
    // 现金为正，重置计数器
    state.company.cashCriticalStreak = 0
    return false
  }

  // 现金 <= 0，检查 4 周内是否有应收款能覆盖缺口
  const shortfall = Math.abs(state.company.cash)
  const incomingReceivables = (state.company.receivables ?? [])
    .filter((r: Receivable) => r.dueInWeeks <= 4)
    .reduce((sum: number, r: Receivable) => sum + r.amount, 0)

  if (incomingReceivables >= shortfall) {
    // 应收款足以覆盖，危机但不致命，发黄色预警
    triggerEmergencyEvent('CASH_WARNING_RECEIVABLE_SAFE', state)
    return false
  }

  // 应收款不够兜底，开始倒计时
  state.company.cashCriticalStreak = (state.company.cashCriticalStreak ?? 0) + 1

  if (state.company.cashCriticalStreak === 1) {
    // 第 1 周：红色警告，AI 给出具体自救选项
    triggerEmergencyEvent('CASH_CRITICAL_START', state)
    return false
  }

  if (state.company.cashCriticalStreak < 4) {
    // 第 2-3 周：持续警告，压力递增
    triggerEmergencyEvent('CASH_CRITICAL_ONGOING', state)
    return false
  }

  if (state.company.cashCriticalStreak === 4) {
    // 第 4 周（满 1 个月）：最后警告 + AI 最终自救方案
    triggerEmergencyEvent('CASH_FINAL_WARNING', state)
    return false  // 给最后 1 周机会
  }

  // 连续 > 4 周（超过 1 个月）仍无法解决：Game Over
  return true
}
```

#### 条件 B：创始人崩溃（FOUNDER_COLLAPSE）
```
触发条件（同时满足）：
  1. bwStress > 98 持续 3 周
  2. 任意属性因 Burnout 惩罚降至 < 20
  
设计意图：单纯的高压不会直接 Game Over，
但永久属性损耗累积到极限才会。
这模拟了真实的"慢性崩溃"而非"瞬间死亡"。
```

```typescript
function checkFounderCollapse(state: GameState): boolean {
  const isBurntOut = state.founder.bwStress > 98 && state.founder.bwStressStreak >= 3
  const hasCollapsedAttribute = Object.values(state.founder.attributes).some(v => v < 20)
  return isBurntOut && hasCollapsedAttribute
}
```

#### 条件 C：市场窗口关闭（MARKET_DEATH）
```
各阶段最大停留时间（超时视为"错过窗口期"）：
  SEED  → MVP   ：最多 26 周（半年）
  MVP   → PMF   ：最多 52 周（一年）
  PMF   → SCALE ：最多 78 周（1.5年）
  SCALE → IPO   ：无强制上限（但有负向事件压力）
  
设计意图：2026 年的市场不等人，
拖太久也是一种失败。
```

```typescript
const STAGE_TIME_LIMITS: Record<CompanyStage, number> = {
  SEED:  26,
  MVP:   52,
  PMF:   78,
  SCALE: Infinity,
  IPO:   Infinity,
  TITAN: Infinity,
}

function checkMarketDeath(state: GameState): boolean {
  const limit = STAGE_TIME_LIMITS[state.company.stage]
  return state.company.weeksInCurrentStage > limit
}
```

#### 条件 D：强制退出（FORCED_EXIT）
```
由特定高阶随机事件直接触发，无法通过数值规避。
示例事件：
  - [TITAN阶段] 反垄断强制解散
  - [IPO阶段] 财务造假被曝光（由连续 FIN 属性 < 30 时触发概率上升）
  - [SCALE阶段] 核心创始人跑路（Bonding 长期低于 20）
```

### 1.3 Game Over 执行流程

```
1. 每次 sprint 结束后调用 checkGameOver(state)
2. 若触发：
   a. 写入 Rehearsal.isFailed = true
   b. 调用 Gemini（高性能模型）生成 AI 尸检报告（failedReason 字符串）
   c. 写入 LeaderboardEntry（Graveyard 模式）
   d. 终端输出死亡动画（屏幕逐行清空效果）
   e. 提示用户：查看复盘 / 带着经验重开
```

## 三、V1：概率事件池（P1 阶段实现）

> **实现标准**：简单、可靠、先跑通。不涉及向量数据库。

### 3.1 触发引擎

```typescript
// src/engine/events/probability-engine.ts

export function pickEventsForSprint(
  state: GameState,
  weeks: number
): GameEvent[] {
  const triggeredEvents: GameEvent[] = []
  const eligibleEvents = loadEventsByStage(state.company.stage)

  for (const event of eligibleEvents) {
    // 冷却期检查
    if (isOnCooldown(event, state)) continue

    // 计算最终触发概率
    const probability = calcFinalProbability(event, state)

    // 每周投一次骰子
    for (let week = 0; week < weeks; week++) {
      if (Math.random() < probability / weeks) {
        triggeredEvents.push(event)
        markCooldown(event, state)
        break  // 同一事件在一个 sprint 内只触发一次
      }
    }
  }

  // 按优先级排序：BLACK_SWAN > FINANCE > PEOPLE > MARKET > TECH
  return sortByCategory(triggeredEvents)
}

function calcFinalProbability(event: GameEvent, state: GameState): number {
  let prob = event.baseProbability

  for (const [attr, config] of Object.entries(event.probabilityModifiers)) {
    const attrValue = state.founder.attributes[attr as keyof FounderAttributes]
    if (attrValue !== undefined && attrValue < config.threshold) {
      prob *= config.multiplier  // 属性低于阈值时概率放大
    }
  }

  return Math.min(prob, 0.95)  // 概率上限 95%，永远保留不确定性
}
```

## 四、V2：语义驱动引擎（P2 阶段升级）

> **实现标准**：用向量相似度替代硬编码概率，事件触发更有"因果感"。
> **前提**：pgvector 已接入，事件 JSON 已预处理生成 embedding。

### 4.1 核心原理

```
传统方式：
  if random() < 0.05 → 触发"技术债事件"

V2 语义方式：
  1. 将当前状态 → 向量（State Embedding）
  2. 在事件库向量空间中做 KNN 搜索
  3. 相似度最高的 Top-K 事件参与候选池
  4. 候选池内再做一次轻量概率筛选（保留随机性）
```

### 4.2 状态向量化


### 4.3 向量门限机制（Vector Thresholding）

防止语义偏移——"技术危机"不应匹配到"财务事件"：

```typescript
async function pickEventsSemantic(
  state: GameState,
  topK: number = 5
): Promise<GameEvent[]> {
  const stateVec = await buildStateEmbedding(state)
  const stageFilter = state.company.stage

  // pgvector KNN 查询，带阶段过滤 + 相似度阈值
  const candidates = await db.$queryRaw`
    SELECT event_id, 1 - (embedding <=> ${stateVec}::vector) AS similarity
    FROM event_embeddings
    WHERE ${stageFilter} = ANY(stages)
      AND 1 - (embedding <=> ${stateVec}::vector) > 0.65  -- 相似度阈值
    ORDER BY embedding <=> ${stateVec}::vector
    LIMIT ${topK}
  `

  // 候选事件再做一次轻概率筛选，保留随机性
  return candidates
    .map(c => loadEventById(c.event_id))
    .filter(e => Math.random() < 0.6)  // 60% 概率从候选中实际触发
}
```

## 五、事件执行层

### 5.1 事件执行器

```typescript
// src/engine/events/executor.ts

export async function executeEvent(
  event: GameEvent,
  state: GameState
): Promise<{ updatedState: GameState; narrative: string }> {
  
  // 1. 应用数值效果（确定性，System 1）
  let updatedState = applyEffects(state, event.effects)
  
  // 2. 检查是否触发 Game Over
  if (event.isGameOver || checkGameOver(updatedState)) {
    return handleGameOver(updatedState, event)
  }
  
  // 3. AI 生成叙事（不确定性，System 2）
  const narrative = await generateEventNarrative(event, updatedState)
  
  // 4. 检查是否触发 Aha-Moment（事件可以作为 Aha 的上下文）
  const ahaMoment = checkAhaMoment(updatedState)
  if (ahaMoment) {
    // Aha-Moment 在 narrative 之后单独渲染
    scheduleAhaMoment(ahaMoment, updatedState)
  }
  
  return { updatedState, narrative }
}

async function generateEventNarrative(event: GameEvent, state: GameState): Promise<string> {
  // 分级模型路由（来自技术重难点文档 2.3）
  const model = event.category === 'BLACK_SWAN' ? 'gemini-2.0-pro' : 'gemini-2.0-flash'
  
  return await cortexAI.generate({
    model,
    stage: state.company.stage,
    eventPrompt: event.narrativePrompt,
    stateContext: buildMinimalContext(state),
  })
}
```

### 5.2 Sprint 主循环集成

```typescript
// src/engine/simulator.ts （关键路径）

export async function runSprint(state: GameState, weeks: number): Promise<SprintResult> {
  let currentState = state
  const sprintLog: SprintLog[] = []

  for (let week = 1; week <= weeks; week++) {
    // 1. 数值步进（System 1，确定性）
    currentState = stepNumerics(currentState)
    
    // 2. 事件抽取（V1 或 V2，由 feature flag 控制）
    const events = USE_SEMANTIC_ENGINE
      ? await pickEventsSemantic(currentState)
      : pickEventsForSprint(currentState, 1)
    
    // 3. 执行事件
    for (const event of events) {
      const { updatedState, narrative } = await executeEvent(event, currentState)
      currentState = updatedState
      sprintLog.push({ week, event: event.name, narrative })
      
      // 若 Game Over，立即中止循环
      if (currentState.isFailed) break
    }
    
    if (currentState.isFailed) break
  }

  // 4. Sprint 结束检查 Aha-Moment
  const aha = checkAhaMoment(currentState)
  
  // 5. Game Over 检查
  const gameOver = checkGameOver(currentState)
  
  return { finalState: currentState, log: sprintLog, ahaMoment: aha, gameOver }
}
```

## 六、事件生产指南（批量生成策略）

根据技术路线图建议，**不要手写事件，用 LLM 批量生成**：

```
生成策略：
  目标：每阶段 30-50 个事件，6 个阶段 = 约 200+ 事件
  
Prompt 模板：
  "你是 lemeone-lab 的首席内容设计师。
   请为 [MVP] 阶段生成 30 个商业事件 JSON。
   
   要求：
   - category 分布：MARKET(10) / TECH(8) / PEOPLE(6) / FINANCE(4) / BLACK_SWAN(2)
   - baseProbability 范围：0.02 - 0.15
   - 每个事件必须有 semanticTags（3-6个）
   - effects 必须有正反两面事件
   - narrativePrompt 要有具体的叙事方向
   
   输出格式：完整的 JSON 数组"
   
人工审核清单：
  ✅ 概率值不超过 0.2
  ✅ effects delta 在合理范围（单次不超过 ±50）
  ✅ stage 白名单正确
  ✅ 无重复 ID
  
运行验证脚本：
  npx ts-node scripts/validate-events.ts
```

## 七、与其他模块的依赖关系

```
随机事件引擎
  ├── 输入 ← simulator.ts（每 sprint 调用）
  ├── 输入 ← GameState（含 MarketContext + Staff[]）
  ├── 输入 ← MarketContext.industry → industryFilter 过滤
  ├── 输入 ← MarketContext.volatility → 缩放事件概率
  ├── 输入 ← Staff[].loyalty → staffLoyalty 修正器
  ├── 输出 → EventEffect（数值变更，被 stepNumerics 消费）
  ├── 输出 → StaffEffect（员工忠诚度/离职，被 staffModule 消费）
  ├── 输出 → narrativePrompt（被 cortex-ai.ts 消费，生成叙事）
  ├── 触发 → checkAhaMoment()（部分事件会升级为 Aha-Moment）
  ├── 触发 → checkGameOver()（黑天鹅事件可直连 Game Over）
  └── 写入 → LeaderboardEntry.failedReason（Game Over 时写 Graveyard）

V2 语义引擎额外依赖：
  ├── pgvector（事件 embedding 存储与 KNN 检索）
  └── scripts/embed-events.ts（离线预处理，非运行时依赖）
```

## 八、各阶段事件风险分级参考

| 阶段 | 主黑天鹅风险 | 典型高概率小事件 | 解锁的新风险 |
